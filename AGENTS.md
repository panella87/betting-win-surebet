# Agent Instructions

## Repository purpose

`betting-win-surebet` is a private, paper-only downstream strategy repository for surebet / complete-set / arbitrage-style research. It consumes versioned contracts, exports, read-only query outputs, and generic paper infrastructure from `betting-win`.

It does not own provider truth.

## Source-of-truth order

Use truth in this order before planning or changing code:

1. current code
2. fresh retained evidence generated by current code
3. `PROJECT_STATUS.md`
4. `docs/repo_status_current.md`
5. `docs/MASTER_PLAN.md`
6. `docs/001_scope_and_boundaries.md`
7. `docs/002_dependency_contract_with_betting_win.md`
8. `docs/012_runbook.md`
9. `docs/operations/autonomous_72h_runbook.md`
10. README and ADRs

Current code beats stale docs. Do not generate new work from old backlog claims without re-checking current code and current status.

## Hard boundaries

Do not implement provider adapters, provider API clients, provider credentials, wallets, signers, token approvals, orders, cancellations, cashouts, redemptions, splits, merges, transactions, live collectors, public signals, affiliate features, or profitability claims.

Do not connect directly to SX, Azuro, Polymarket, Limitless, or any accepted future provider.

Do not write `betting-win` `core.*` migrations.

Do not vendor generated `betting-win` contracts manually. Consume a pinned `betting-win` contract/export package or fixture bundle only after Federico provides it.

## First lane

```text
polymarket_standard_binary_complete_set_v0
Polymarket standard-binary same-venue complete-set paper arbitrage
```

This is not approved for live use. It is a paper-only research lane. Cross-venue, back/lay, synthetic equivalence, smart routing, market making, and copy trading are parked for v0.

## Acceptance rules

A candidate is not a surebet unless it has exact event and market identity, rule/result-source/protocol-generation equivalence, complete terminal-scenario payoff coverage, executable quote/depth evidence from `betting-win`, outcome-specific fee and cost cash flows, capacity-constrained stake vector after rounding, quote freshness, group-level leg-completion simulation, residual-exposure analysis, and settlement replay.

Reciprocal odds alone are not acceptance evidence.

## Implementation discipline

- Implement one bounded slice per cycle. Continue with `CONTINUE_REQUIRED=yes` while `docs/014_sure_001_remaining_hardening_backlog.md` or `docs/015_local_engine_implementation_backlog.md` still contains safe unchecked work.
- SURE-001 docs/tooling/validators are complete.
- Federico asked for the maximum safe implementation possible; continue through `docs/015_local_engine_implementation_backlog.md`.
- Local-only deterministic solver math, simulation state machines, settlement replay consumers, and private report assembly are allowed only against fake/local fixtures and must not claim real upstream readiness.
- Do not implement provider ingestion or execution paths.
- Do not weaken validators to make a run green.
- Do not add placeholders that silently pass as evidence.
- Do not print secrets or mutate `.env`.
- Do not commit, push, pull, reset, clean, stash, or rewrite branches from inside autonomous implementation cycles.

## Validation

Every task must run:

```bash
npm run validate
```

No task may pass by weakening validators or replacing evidence with mocks.
